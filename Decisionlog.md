### D-001 · STT 엔진·인터페이스·범위 확정
**상황**
- 동영상/오디오를 텍스트로 추출하는 신규 프로젝트. 셋업 전 `PLAN.md`에서 사용자와 방향을 확정함.

**검토한 선택지**
- STT: 클라우드 API(유료·온라인) vs 로컬 Whisper(무료·오프라인)
- 인터페이스: CLI vs GUI
- 미디어 디코딩: FFmpeg 설치 vs faster-whisper 내장 PyAV

**결정**
- STT = 로컬 faster-whisper, 인터페이스 = CLI, 언어 = 한국어 위주(auto 지원), 출력 = txt·md·srt + 화자 구분(선택), 디코딩 = PyAV(FFmpeg 미설치).

**근거**
- 무료·오프라인·프라이버시. RTX 4090으로 large-v3 GPU 추론 가능. PyAV로 FFmpeg 설치 부담 제거.

**영향**
- ctranslate2의 CUDA 12 + cuDNN 9 DLL 의존성 관리 필요(리스크). 화자 구분은 pyannote 게이트 모델 → HuggingFace 토큰 필요.

**재검토 조건**
- GPU DLL 문제로 GPU 사용이 반복 실패하거나, 화자 구분 품질/설치 부담이 과도할 경우 범위 재조정.

### D-002 · 코드 배치 구조 결정 (src/ 대신 패키지)
**상황**
- 표준 셋업 구조는 `src/`를 두지만, PLAN.md는 루트 `audio_to_text/` 패키지 + `transcribe.py` 진입점을 사용.

**검토한 선택지**
- (a) `src/audio_to_text/` 로 이동 vs (b) PLAN.md대로 루트 패키지 유지

**결정**
- (b) 루트 `audio_to_text/` 패키지 + `transcribe.py` 유지. `src/` 폴더는 만들지 않음.

**근거**
- 파이썬 CLI에서 관용적이고 `python transcribe.py`로 바로 실행하기 쉬움. 이미 설계·합의된 구조.

**영향**
- CLAUDE.md 3절에 "`src/` 역할은 `audio_to_text/` 패키지가 대신함"을 명시.

**재검토 조건**
- 배포(pip 패키징)를 정식화할 때 `src/` 레이아웃 재검토.

### D-003 · 기준 계약 문서 미확정 항목 확정
**상황**
- 문서 검토(W-004)에서 계약 문서의 `확인 필요` 2건과 스펙 공백(폴더 출력 구조, SRT 규칙, 화자 번호, 종료 코드)을 발견. 구현(S1) 착수 전 확정 필요.

**검토한 선택지**
- --diarize 준비 안 됨: (a) 안내 후 텍스트만 계속 vs (b) 즉시 중단
- 무음/인식 0건: (a) 빈 출력 파일 생성 vs (b) 파일 미생성 + 경고
- 폴더 입력 + --output-dir: (a) 하위 구조 미러링 vs (b) 평탄화

**결정** (2026-07-04, 사용자 확인)
- --diarize 실패 시 안내 후 **텍스트 변환 계속**(배치 중단 없음).
- 무음 파일도 `(인식된 음성 없음)` 명시한 **출력 파일 생성**, "성공"으로 집계.
- 폴더 출력은 **하위 구조 미러링**.
- (관례 확정) SRT는 v1에서 Whisper 세그먼트 그대로 / 화자 번호는 첫 등장 순 / 종료 코드 0·1·2 / 테스트 오디오는 커밋하지 않고 재생성 / PLAN.md와 충돌 시 계약 문서 우선.

**근거**
- 배치 처리의 연속성(부분 실패가 전체를 막지 않음), 처리 추적 용이성, 이름 충돌 원천 차단, 자동화 연계(종료 코드).

**영향**
- files.py(경로 미러링·충돌 처리), cli.py(종료 코드·diarize 폴백), formatters.py(빈 결과 문구) 구현 방향 고정.

**재검토 조건**
- 실사용에서 빈 출력 파일이 소음이 되거나, 화자 구분을 필수로 쓰는 워크플로가 생기면 재검토.

### D-004 · 데스크톱 GUI 추가 (범위 변경) 및 PySide6 선택
**상황**
- CLI 완성·검증 후(S0~S4, 실사용 W-011) 사용자가 "매번 명령어 입력이 힘들다"며 로컬 데스크톱 앱을 요청. GUI는 PLAN.md에서 "범위 제외(추후 확장 후보)"였던 항목.

**검토한 선택지**
- (a) Tkinter(내장, 소박한 외관) vs (b) PySide6/Qt(현대적 외관, +80MB) vs (c) bat 드래그&드롭만 vs (d) PyInstaller 단일 exe(torch 포함 수 GB)

**결정** (2026-07-04, 사용자 선택)
- **(b) PySide6 GUI**. 구조는 **검증된 CLI(transcribe.py)를 하위 프로세스로 호출하는 얇은 껍데기** — 변환 엔진을 GUI에 재구현하지 않는다.
- 실행은 venv + 더블클릭 런처(bat/바로가기). exe 패키징(d)은 용량 문제로 보류.

**근거**
- 사용자가 외관을 우선해 PySide6 선택. CLI 재사용 구조는 검증 자산을 그대로 활용해 회귀 위험이 최소.

**영향**
- requirements-gui.txt, gui.py, 런처 추가. implementation-plan에 S6 단계 신설. 계약 문서에 "GUI는 CLI 규칙을 재정의하지 않는다" 명시.

**재검토 조건**
- 트레이 상주·변환 큐 관리 등 요구가 커지면 in-process 엔진 구조 재평가. 배포 필요성이 생기면 exe 패키징 재검토.

### D-005 · md 문단 규칙에 최대 길이(60초) 캡 추가
**상황**
- 실사용(W-015)에서 편집된 인터뷰 영상의 md가 7.5분 전체 **문단 1개**로 출력됨. 편집 영상은 발화 사이 쉼이 잘려 있어 기존 기준(간격 ≥2초)만으로는 문단이 나뉘지 않음.

**검토한 선택지**
- (a) 현행 유지 vs (b) 문단 최대 길이(초) 캡 vs (c) 문장 수 기준 분할

**결정**
- (b) 간격 규칙 유지 + **한 문단이 60초를 넘으면 세그먼트 경계에서 강제 분할**.

**근거**
- 단순·결정적이며 타임스탬프의 탐색 유용성을 회복. (c)는 언어별 문장 분리 복잡성 대비 이득이 작음.

**영향**
- formatters.merge_paragraphs(md와 txt --no-timestamps 공용), 계약 "출력 형식" 갱신.

**재검토 조건**
- 실사용 가독성 피드백에 따라 캡 값(60초) 조정.
