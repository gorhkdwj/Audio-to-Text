# 검증 계획 · Audio_to_Text

기능을 만들 때마다 영향 범위에 맞게 검증한다. 검증하지 못한 부분은 숨기지 않고 `미검증 범위`에 적는다. 실패한 테스트를 삭제·완화하지 않고 원인을 수정한다.

## 검증 항목
- 단위 테스트: 타임스탬프 포맷 변환(HH:MM:SS / SRT 밀리초), 출력 경로 계산, 지원 확장자 판별, 화자-세그먼트 매칭 로직.
- 통합 테스트: 오디오 1개 → txt·md·srt 생성 전체 파이프라인.
- 입력 검증: 존재하지 않는 경로, 지원하지 않는 확장자, 폴더 재귀 수집.
- 오류 상황 검증: CUDA DLL 실패 시 CPU 폴백, `--diarize` 미설치/토큰 없음 안내.
- 경계값 검증: 무음/매우 짧은 파일, 긴 파일 진행률.
- 문서-코드 정합성: requirements-contract.md의 포맷 규칙과 formatters.py 출력 일치.
- README-기능 일치: README에 적은 명령이 실제로 동작.
- 종료 코드 검증: 성공(0)/일부 실패(1)/인자 오류(2)가 계약대로 반환되는지 확인.

## STT 통과 판정 기준
STT 출력은 정답 문장과 100% 일치하지 않는 것이 정상이다(모델 특성). 다음 기준으로 판정한다:
- **통과**: 정답 문장의 핵심 단어(내용어)가 모두 포함되고 의미가 보존됨. 조사·띄어쓰기·문장부호·숫자 표기 차이는 허용.
- **실패**: 핵심 단어 누락, 의미가 달라지는 오인식, 타임스탬프가 실제 발화 구간과 명백히 어긋남.
- 판정은 육안 대조로 하되, 사용한 정답 문장과 판정 결과를 Worklog에 기록한다.

## 검증 방법과 도구
- **테스트 오디오 생성**(`tools/`): PowerShell `System.Speech`(TTS)로 "알고 있는 정답 문장" WAV 생성. 화자 구분용은 서로 다른 두 보이스를 이어 붙인 WAV.
- 테스트 오디오/동영상은 저장소에 **커밋하지 않고** `tools/` 스크립트로 필요할 때 재생성한다(`.gitignore`의 미디어 제외 규칙과 정합).
- **테스트 동영상 생성**: PyAV로 WAV를 m4a/mp4로 변환해 동영상 입력 경로 검증.
- 실행 검증: `python transcribe.py test.wav --format txt md srt` → 3파일의 타임스탬프·내용 확인.
- 디바이스 검증: 콘솔 로그로 GPU(cuda/float16) 사용 여부 확인, `--device cpu`로 폴백 확인.
- 옵션 검증: 폴더 일괄 처리, `--no-timestamps`, `--overwrite` 동작 확인.
- 화자 구분: HF 토큰 준비 시 2-보이스 파일로 라벨 확인, 없으면 안내 메시지까지만 확인.

## 미검증 범위 (2026-07-04, S4 완료 기준)
- `avi` 컨테이너: PyAV로 AAC-in-AVI 표본을 만들기 부자연스러워 실측 보류(비주류 조합). 실사용 avi 파일로 확인 권장.
- 오디오 확장자 중 flac·ogg·opus·aac·wma: PyAV 지원 범위이나 표본 미생성으로 미실측(wav·mp3·m4a는 실측 완료).
- 실사용 미디어(방언·잡음·장시간) 인식 정확도: 별도 표본 필요.
- 화자 구분: 3인 이상·장시간 대화 미실측(2인·3턴 검증 완료). pyannote의 GPU 가속 사용 여부는 별도 계측 안 함(기능 검증만).

## 확인된 한계 (버그 아님, 사용상 주의)
- **다국어 혼용 파일**: Whisper는 파일당 한 언어로 디코딩하므로, 한국어+영어 혼합 파일에서 영어 발화가 통째로 누락되고 인접 세그먼트의 타임스탬프도 부정확해질 수 있음(S4에서 확인). 화자 구분 포함 모든 변환은 단일 언어 파일 사용을 권장. README에 명시.

## 검증 이력 메모
- (S4, W-010) 화자 구분 실측 통과: 2인 영어 대화(David→Zira→David)에서 `화자 1`/`화자 2` 라벨 분리, 첫 등장 순서 번호(D-003), `--num-speakers 2` 힌트 — 모두 정확. GPU Whisper(large-v3)와 pyannote 동일 프로세스 공존 확인(T-004 해결 후). 미준비 시 안내 후 계속(D-003 폴백)도 T-003 과정에서 실전 확인됨. 의존성 이슈 3건은 T-002~T-004 참조.
- (S3, W-009) 폴더 일괄+하위 구조 미러링, `--no-timestamps`, `--language auto`(영어 `en` 감지), 무음 파일 `(인식된 음성 없음)` 출력, 손상 파일 혼합 배치 → 성공 1/실패 1 + **종료 코드 1**, 컨테이너 디코딩(동영상 mkv·mov·webm(Opus)·ts + 오디오 mp3·m4a) — 전부 E2E 통과 (small/cpu).
- (S2, W-007) TTS 픽스처 한계: 문장 1의 "화창합니다"를 small·medium·large-v3 세 모델이 동일하게 오인식("확장/확창") → 파이프라인 결함이 아닌 기계음 발음 한계로 판정. 문장 2는 숫자 표기 차이("세 가지"→"3가지", 허용)만으로 통과.
- 참고: 파일 리다이렉트 캡처 시 [실패](stderr)가 stdout보다 먼저 보일 수 있음 — 버퍼링 차이로 인한 표시 순서 문제이며 실제 콘솔(TTY)에서는 정상 순서.
