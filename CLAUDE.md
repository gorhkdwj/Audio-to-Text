# CLAUDE.md · Audio_to_Text 작업 헌법

이 문서는 이 프로젝트에서 AI 협업 에이전트가 **매번 지시하지 않아도 항상 따라야 하는 기준**이다. 사용자와 에이전트 모두 이 규칙을 기준으로 일한다. 규칙이 바뀌면 이 문서를 먼저 갱신하고 Decisionlog에 이유를 남긴다.

## 0. 프로젝트 정보
- 프로젝트 이름: Audio_to_Text (동영상/오디오 → 텍스트 추출 CLI)
- 목적: 동영상 또는 오디오 파일을 입력하면 음성을 인식(STT, Speech-to-Text = 음성을 글자로 바꾸는 기술)해 txt/md/srt 파일로 텍스트를 추출한다.
- 주요 사용자: 본인(개발자)이 직접 사용하는 CLI 도구. 단, 개발 기록과 문서화는 남이 봐도 이해할 수준으로 상세하게 남긴다.
- 최종 산출물: 명령줄(CLI) 자동화 도구 (`python transcribe.py <입력> [옵션]`)
- 마감·제약: 마감일 없음. Python 3.10.8 / Windows 11 / NVIDIA RTX 4090(24GB). FFmpeg 미설치 → faster-whisper 내장 PyAV로 디코딩(FFmpeg 설치 불필요).
- 현재 사용자 수준: 초급자도 이해할 수 있게 설명한다(용어 풀이 포함).
- 원하는 진행 방식: 초급자도 이해하게 + 꼼꼼한 문서화 + 검증 중시. 각 단계를 실제 실행/테스트로 확인하고 기록을 상세히 남긴다.

## 1. 대화와 보고 방식
- 모든 응답은 격식있는 한국어 대화체(합쇼체)로 작성한다.
- 결론을 먼저, 근거·세부는 뒤에.
- 기술 용어는 처음 등장할 때 한 줄로 설명한다. (사용자 수준: 초급자 기준)
- 불확실하면 단정하지 말고 `확인 필요`로 표기한다.
- 실행/확인하지 않은 것을 완료라고 말하지 않는다.

## 2. 작업 시작 전 확인
- 현재 폴더 구조와 기존 파일을 먼저 확인한다.
- 기존 파일·변경사항(특히 `PLAN.md`)을 함부로 덮어쓰지 않는다.
- 바로 구현하지 말고, 영향 범위와 계획을 먼저 정리한다.

## 3. 파일 구조 원칙
- 이 프로젝트는 파이썬 CLI 패키지 형태다. `src/` 역할은 루트의 **`audio_to_text/` 패키지 + `transcribe.py` 진입점**이 대신한다(파이썬 CLI 관용 구조, `docs/project-plan.md` 참조).
- `tests/` 테스트 · `tools/` 개발·검증 보조 스크립트(예: 테스트용 오디오 생성) · `docs/` 문서 · `out/` 실행·임시 산출물(Git 제외).
- 원본/가공 데이터셋이 없으므로 `data/`는 만들지 않는다. 필요해지면 그때 추가한다.
- 사용자가 직접 실행하는 핵심 기능은 패키지 코드. 개발자 전용 보조 스크립트는 `tools/`. 반복 실행 결과·변환 출력물은 `out/`.
- 필요 없는 폴더는 만들지 않고 이유를 남긴다.

## 4. 구현 원칙
- 처음부터 전체를 달리지 않고 가장 작은 성공 단위부터 만든다. (1차: 기본 txt/md/srt 변환 → 2차: GPU 확인 → 3차: 화자 구분)
- 각 단계는 다음을 명확히 한다: 왜 필요한가 / 이전 단계와 연결 / 다음 단계로 전달 / 완료 조건 / 검증 방법 / 실패 시 중단점.
- 단계 완료 보고에 포함: 전체 중 현재 위치, 이번에 한 일, 변경 파일, 검증 결과, 남은 단계, 예상 잔여량, 발생한 문제·T-ID, 필요 시 Git commit/push 여부.

## 5. 기준 계약 문서 원칙
- 구현 전 `docs/requirements-contract.md`를 확정한다: 입력 형식 / 출력 형식 / 지원 확장자 / 타임스탬프·SRT 형식 / 화자 라벨 규칙 / 오류 판정 / 폴백 규칙 / 완료 기준.
- 같은 규칙(예: 타임스탬프 포맷, 화자 라벨)이 파일마다 다르게 정의되지 않게 한다. 기준이 바뀌면 코드보다 계약 문서를 먼저 갱신하고 이유를 기록한다.

## 6. 테스트와 검증 원칙
- 영향 범위에 맞춰 검증한다: 단위·통합·입력·오류 상황·경계값 검증, 문서-코드 정합성, README-기능 일치.
- STT 결과는 TTS(기계 음성 합성)로 만든 테스트 파일로 "알고 있는 정답 문장"과 대조해 검증한다.
- 검증하지 못한 부분은 숨기지 말고 `미검증 범위`로 보고한다. 실패한 테스트를 삭제·완화해 통과시키지 않고 원인을 수정한다.

## 7. 보안·개인정보·비밀정보 원칙
- 다음을 코드·문서·로그·테스트 데이터에 넣지 않는다: API 키, 비밀번호, 액세스 토큰(특히 **HuggingFace 토큰**), 세션 쿠키, 실제 개인정보.
- HuggingFace 토큰은 `HF_TOKEN` 환경변수 또는 `--hf-token` 인자로만 받고, 코드/문서/로그에 값 자체를 남기지 않는다.
- 사용자가 변환하려는 원본 오디오/동영상은 수정하지 않는다. 출력은 별도 파일로 만든다.
- 비밀정보가 발견되면 임의로 로그를 편집하지 말고, 노출 범위와 안전한 교체·폐기 방법을 사용자에게 보고한다.

## 8. 문서화 원칙
- README에는 실제 구현된 기능만 쓴다. 구현되지 않은 기능(예: 화자 구분)을 완성 전에 제공한다고 표현하지 않는다.
- `PLAN.md`(초기 설계), 기준 계약 문서, 코드, 테스트, README, Worklog, Decisionlog가 서로 모순되지 않게 관리한다.

## 9. 외부 자료 사용 원칙
- 외부 자료(모델 카드, 라이브러리 문서 등) 사용 시 기록한다: URL / 제목 / 확인일 / 가져온 판단 근거 / 적용 범위.
- 공개 자료로 확인할 수 없는 사실은 단정하지 않는다.

## 10. Git 원칙
- 아직 git 저장소가 아니다. 첫 커밋 전 사용자에게 `git init` 여부를 확인한다.
- 주요 단계가 끝날 때마다: 변경 파일 확인 → 불필요한 파일 제외 확인 → 테스트/검증 실행 → commit → push.
- 로그, 비밀정보(HF 토큰), `.venv/`, 모델 캐시, 변환 출력물(`out/`), 대용량 미디어 파일은 Git에 올리지 않는다.

## 11. 로그 기록 규칙

### Worklog.md — 모든 작업마다
**모든 작업을 빠짐없이 기록한다.** 주요 요청뿐 아니라 설정 변경, git 셋업/커밋/푸시, 문서 수정, 사소한 조정, 설명·조사 작업까지 실제로 수행한 일이면 모두 남긴다. 규모가 작아도 생략하지 않는다.
- 한 번의 사용자 요청에서 성격이 다른 여러 작업을 했다면 각각 별도 W-ID로 남기거나, 하나의 W-ID 안에 항목으로 나눠 적는다.
- 작업을 수행했는데 아직 Worklog에 없으면, 다음 작업 전에 소급해서라도 기록한다.
- ID는 W-001부터 순번으로 증가시키며, 최신 항목을 위에 추가한다.
```
### W-00N · 작업 제목
**요청** / **수행 작업** / **변경 파일** / **검증**(미실행 시 사유) / **판단 근거** / **결과**(완료·남은 작업)
```

### Decisionlog.md — 방향을 바꾸는 결정만
대상: 문제 선정, 기술 스택 선택, 구현 범위 확정·축소, 데이터 처리 방식, 보안·개인정보 처리, 테스트·검증 방식, 배포·제출·공유 방식, 큰 구조 변경. 사소한 수정은 Worklog만.
```
### D-001 · 결정 제목
**상황** / **검토한 선택지** / **결정** / **근거** / **영향** / **재검토 조건**
```

### Troubleshootinglog.md — 실제 문제 발생 시
실제 오류·실패·환경 문제(특히 **CUDA/cuDNN DLL 로드 실패**)·검증 실패·설계 충돌이 발생하면 기록한다. 같은 문제가 반복되면 새 T-ID를 만들기 전에 기존 T-ID를 먼저 확인한다.
```
### T-001 · 문제 제목
**발생 상황** / **증상** / **확인된 원인**(+불확실점) / **조치**(+최종 해결) / **재발 방지**
```

## 12. 완료 보고 기준
작업 완료 시 다음 형식으로 보고한다. 실행하지 않은 검증은 완료라고 하지 않는다.
```
완료했습니다.
- 수행한 작업:
- 변경 파일:
- 검증 결과:
- 기록: Worklog / Decisionlog / Troubleshootinglog:
- 남은 작업:
- 한계 또는 미검증 범위:
- Git 상태:
```
